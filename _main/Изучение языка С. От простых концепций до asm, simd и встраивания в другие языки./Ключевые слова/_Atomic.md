# _Atomic

**Категория:** Квалификатор типа (C11)

## Назначение

Объявление атомарного типа для безопасной многопоточности. Атомарный - от слова атом (Неделимый, неизменяемый). Атомарные операции объявляются подключением заголовка stdatomic.h.

Разберём простейший пример - изменение одной области памяти двумя потоками. По-сути, любая операция состоит из шагов:
1. Прочитать значение по адресу из памяти в рабочую область (Регистры процессора).
2. Модифицировать значение на целевое в регистре.
3. Записать из регистра по исходному адресу в памяти.

Если несколько потоков будут одновременно получать доступ к памяти и производить вычисления, пусть даже и одинаковые, подобное поведение может привести к следующим проблемам:
1. SegFault при работе с массивами и выходе за их пределы. Адрес в памяти чётко зарезервирован под структуру и не может быть динамически расширен. Оставшееся пространство заполняется пустыми данными (Мусор) для соблюдения размерности массива и выравнивания памяти.
2. При модификации значения int или другого типа, не являющегося массивом - Undedefined behavior. Неизвестно, какой поток первым и последним запишет значение из регистра по адресу.
3. Мусорные значения при работе с байтами. Вместо нормальных 10 или 100 в байтовой записи, получим нечленоразборный мусор: \xfa или другое значение, которое записано по адресу но не является целевым (Т.е. является мусором).

Как атомарные операции и типы решают подобные проблемы? Если thread_local давал возможность скопировать данные для каждого потока из адреса памяти, но при изменении по адресу - всё равно записывался мусор, атомарные - решают подобные проблемы "Неделимостью" процесса модификации адреса - вместо 3 операций по чтению, модификации числа и записью мы получаем ОДНУ операцию (с блокировкой шины процессора для сложных операций), которая гарантирует что исполнение процедуры случится здесь и сейчас. Обеспечивается следующим образом:
1. Аппаратная поддержка. Процессоры имеют операции, которые исполняются за 1 такт (Вместо 3 как выше!) и блокируют шину памяти через префикс LOCK (x86 only):
```assembler
; Обычный инкремент (НЕ атомарный)
mov eax, [counter]  ; Чтение из памяти
add eax, 1          ; Изменение в регистре
mov [counter], eax  ; Запись в память
; Между этими инструкциями может вмешаться другой поток!

; Атомарный инкремент (x86)
lock inc [counter]  ; ОДНА инструкция!
; Префикс lock блокирует шину памяти для других процессоров/ядер
```
Это работает следующим образом:
LOCK посылает шине процессора сигнал о блокировке области (Адреса) памяти.
Протоколы Cache Coherence Protocol (MESI\MOESI, подробнее: [[MESI, MOESI]]) - обеспечивают согласованность (Когерентность) кэша, что обеспечивает доступ ядер к согласованному состоянию памяти.
2. Стратегии атомарности.
		1. Блокировка шины (Уже рассмотрено)
		2. CAS (Compare And Swap)
			CAS (Сравнение-с-Обменом) — это атомарная операция, которая лежит в основе всех сложных lock-free алгоритмов. CAS принимает **три аргумента**: **Адрес** ячейки памяти (`ptr`) **Ожидаемое значение** (`expected`) **Новое значение** (`desired`). Операция сравнивает значение по адресу с `expected`: если равны — записывает `desired` и возвращает `true`, иначе — записывает текущее значение в `expected` и возвращает `false`. **Важно:** CAS не гарантирует прогресс — при высокой конкуренции поток может бесконечно циклить (live lock), постоянно получая `false`. Не требует блокировки шины - быстрее, но за счёт многоразовых повторов не является эталоном производительности.
## Пример

```c
#include <stdatomic.h>

_Atomic int counter = 0;
atomic_int count = 0;  // макрос из stdatomic.h

// Атомарные операции
atomic_fetch_add(&count, 1);    // count++
atomic_fetch_sub(&count, 1);    // count--
int val = atomic_load(&count);  // чтение
atomic_store(&count, 42);       // запись
```
### Реальный пример: CAS-цикл

Самый частый паттерн использования — **CAS-цикл** (или "оптимистичная блокировка"). Ищи в _resources/base_level_C/atomic.

## Применение

- Счётчики
- Флаги
- Lock-free структуры данных

### Примечание:

При низкой конкурентности - CAS выигрывает по производительности, но при увеличении конкурентных потоков - деградирует. Так же CAS подвержен ABA - pop -> A, push -> B, push -> A (тот же адрес, но данные другие!). Такое происходит из-за того, что CAS смотрит на указатель а не на значение переменной. Tagged Pointer - счётчик, который является решением проблемы (То есть считаем, если счётчик не тронут - и в правду одинаковые данные. Иначе - невалидные данные).

## Пример из практики: CAS-цикл

Файл: `_resources/base_level_C/atomic/atomic.c`

```c
#include <stdatomic.h>

atomic_int global_value;
atomic_int min_value;

void atomic_multiply(atomic_int* val, int multiplier) {
    int old = atomic_load(val);
    int new_val;
    
    do {
        new_val = old * multiplier;
    } while (!atomic_compare_exchange_strong(val, &old, new_val));
    // CAS-цикл: повторяем пока не удастся атомарно обновить
}

void atomic_min(atomic_int* val, int candidate) {
    int old = atomic_load(val);
    
    while (candidate < old) {
        if (atomic_compare_exchange_strong(val, &old, candidate)) {
            break;  // Успешно установили минимум
        }
    }
}
```
